# 2. 需求描述

## 2.1 系统概述

本系统为**需求管理系统**，是一个面向软件工程团队的轻量级需求全生命周期管理平台。系统围绕"产品 → 项目 → 需求"三级管理模型，提供从需求定义、测试追踪、缺陷管理到里程碑基线与分支变更的完整工作流支持，并配备协同评论和审计日志功能以保障团队协作与操作可追溯性。

- **技术架构**：前后端分离，后端采用 Python FastAPI 框架，前端采用 Vue.js 3 + Vite 构建，数据库使用 PostgreSQL。
- **交互方式**：图形化 Web 界面（GUI），提供数据库可视化管理、数据增删改查、SQL 查询编辑器、统计仪表板及性能分析实验室等模块。

## 2.2 功能性需求

### FR-1 数据库连接管理

| 编号   | 需求描述                                                                     |
| ------ | ---------------------------------------------------------------------------- |
| FR-1.1 | 系统启动时自动尝试连接默认配置的 PostgreSQL 数据库，支持最多 10 次重试       |
| FR-1.2 | 提供连接配置界面，允许用户手动输入主机、端口、数据库名、用户名、密码进行连接 |
| FR-1.3 | 支持查看当前连接状态，并提供断开连接功能                                     |
| FR-1.4 | 连接成功后自动加载数据库元数据（Schema 列表与各 Schema 下的表列表）          |

### FR-2 产品管理

| 编号   | 需求描述                                                                          |
| ------ | --------------------------------------------------------------------------------- |
| FR-2.1 | 支持产品的创建、查看、编辑和删除（CRUD）                                          |
| FR-2.2 | 产品信息包括名称（全局唯一）、描述、状态（active/archived）、路线图、版本号和标签 |
| FR-2.3 | 支持产品成员管理，可为产品添加成员并分配角色（owner/admin/member/viewer）         |
| FR-2.4 | 同一用户在同一产品中仅能拥有一个角色（唯一约束）                                  |

### FR-3 项目管理

| 编号   | 需求描述                                                                  |
| ------ | ------------------------------------------------------------------------- |
| FR-3.1 | 支持项目的创建、查看、编辑和删除（CRUD）                                  |
| FR-3.2 | 项目可归属于某个产品（可选），当产品被删除时项目的产品关联自动置空        |
| FR-3.3 | 项目信息包括名称（全局唯一）、描述、状态（active/archived）和当前会话标识 |
| FR-3.4 | 支持项目成员管理，可为项目添加成员并分配角色（owner/admin/member/viewer） |

### FR-4 需求管理

| 编号   | 需求描述                                                                                                           |
| ------ | ------------------------------------------------------------------------------------------------------------------ |
| FR-4.1 | 支持需求的创建、查看、编辑和软删除（通过 deleted 标记）                                                            |
| FR-4.2 | 需求采用三级树形结构：顶层需求（top_level）→ 低层需求（low_level）→ 任务（task）                                   |
| FR-4.3 | 需求具有六种状态流转：draft → under_review → confirmed → in_progress → completed → archived                        |
| FR-4.4 | 需求支持优先级设置（low/medium/high）、负责人指派、截止日期和标签管理                                              |
| FR-4.5 | 支持需求之间的平行关联关系，包括阻塞（blocks）、依赖（depends_on）、关联（relates_to）和重复（duplicates）四种类型 |
| FR-4.6 | 需求的父子关系严格遵循层级规则：顶层需求无父需求；低层需求的父需求必须是顶层需求；任务的父需求必须是顶层或低层需求 |
| FR-4.7 | 禁止需求形成循环父子引用关系（通过递归 CTE 检测）                                                                  |
| FR-4.8 | 支持自定义字段扩展（custom_fields，JSONB 格式）                                                                    |

### FR-5 测试用例管理

| 编号   | 需求描述                                                      |
| ------ | ------------------------------------------------------------- |
| FR-5.1 | 支持测试用例的创建、查看和状态管理（draft/active/deprecated） |
| FR-5.2 | 测试用例归属于项目，包含标题、描述、来源和创建人信息          |
| FR-5.3 | 支持需求与测试用例的多对多关联，通过关联表记录验证关系        |
| FR-5.4 | 关联的需求与测试用例必须属于同一项目（跨项目校验）            |

### FR-6 缺陷管理

| 编号   | 需求描述                                                                                                                          |
| ------ | --------------------------------------------------------------------------------------------------------------------------------- |
| FR-6.1 | 支持缺陷的创建、查看、编辑和状态流转（open → in_progress → resolved → verified → closed / rejected）                              |
| FR-6.2 | 缺陷具有严重程度（critical/high/medium/low）和优先级（P0/P1/P2/P3）双维度分级                                                     |
| FR-6.3 | 每个缺陷必须关联一个需求，且缺陷与需求必须属于同一项目（通过复合外键保证）                                                        |
| FR-6.4 | 缺陷支持多角色指派：报告人（reporter）、开发负责人（dev_assignee）、测试负责人（tester_assignee）和当前负责人（current_assignee） |

### FR-7 里程碑管理

| 编号   | 需求描述                                                                                         |
| ------ | ------------------------------------------------------------------------------------------------ |
| FR-7.1 | 支持里程碑的创建和管理，类型包括常规（regular）、基线（baseline）、分支（branch）和合并（merge） |
| FR-7.2 | 里程碑可标记为基线，支持 Sprint 和版本号的记录                                                   |
| FR-7.3 | 里程碑支持需求快照功能，将某一时刻的需求状态保存为里程碑节点（snapshot）                         |
| FR-7.4 | 里程碑节点记录需求的完整快照数据，包括类型、状态、标题、描述、父需求及排序信息                   |

### FR-8 分支与变更管理

| 编号   | 需求描述                                                                                        |
| ------ | ----------------------------------------------------------------------------------------------- |
| FR-8.1 | 支持基于里程碑创建分支，分支具有四种状态：active → under_review → merged / closed               |
| FR-8.2 | 同一项目内分支名称唯一                                                                          |
| FR-8.3 | 分支下可记录变更集，变更类型包括新增（added）、修改（modified）、删除（deleted）和移动（moved） |
| FR-8.4 | 变更集记录变更前后的完整数据（before_data / after_data），支持差异对比                          |
| FR-8.5 | 删除里程碑时，若仍有分支以其为基线，则拒绝删除（RESTRICT 策略）                                 |

### FR-9 协同评论

| 编号   | 需求描述                                                       |
| ------ | -------------------------------------------------------------- |
| FR-9.1 | 支持对需求、缺陷、测试用例和里程碑四类对象添加评论（多态关联） |
| FR-9.2 | 评论支持树形回复结构（通过 reply_to_id 自引用）                |
| FR-9.3 | 评论仅在同一项目范围内有效，禁止跨项目评论和跨项目回复         |
| FR-9.4 | 评论支持软删除                                                 |

### FR-10 审计日志

| 编号    | 需求描述                                                                 |
| ------- | ------------------------------------------------------------------------ |
| FR-10.1 | 系统记录所有关键操作的审计日志，包括操作人、操作类型、目标对象和详细数据 |
| FR-10.2 | 审计日志可同时关联产品和项目，并校验两者的归属一致性                     |
| FR-10.3 | 审计日志通过多态引用（target_type + target_id）记录对任意业务对象的操作  |

### FR-11 SQL 查询编辑器

| 编号    | 需求描述                                                 |
| ------- | -------------------------------------------------------- |
| FR-11.1 | 提供 SQL 查询编辑器界面，允许用户输入并执行任意 SQL 语句 |
| FR-11.2 | 查询结果以结构化表格形式展示                             |

### FR-12 数据表浏览

| 编号    | 需求描述                                 |
| ------- | ---------------------------------------- |
| FR-12.1 | 提供侧边栏浏览数据库的 Schema 和表结构   |
| FR-12.2 | 支持查看表的字段定义、数据类型和约束信息 |
| FR-12.3 | 支持以分页表格形式浏览表中的实际数据     |

### FR-13 项目统计仪表板

| 编号    | 需求描述                                                                           |
| ------- | ---------------------------------------------------------------------------------- |
| FR-13.1 | 提供项目级统计视图，展示需求总数、各状态分布、缺陷数、测试用例数、里程碑数和分支数 |
| FR-13.2 | 自动计算并展示需求完成率                                                           |
| FR-13.3 | 统计数据通过数据库视图（v_project_statistics）实现，支持高效聚合查询               |

### FR-14 复杂查询功能

| 编号    | 需求描述                                                                                                                                                                                         |
| ------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| FR-14.1 | 需求追溯查询（fn_requirement_trace）：连接 requirements、projects、requirement_test_links、test_cases、defects 共 5 张表，查询某项目下所有需求的完整追溯信息，包括关联测试用例和缺陷的 JSON 聚合 |
| FR-14.2 | 项目进度统计查询（fn_project_progress）：使用 4 个 CTE（requirement_stats、defect_stats、test_coverage_stats、milestone_stats）进行多层嵌套聚合，计算需求完成率和测试覆盖率                      |
| FR-14.3 | 里程碑交付风险查询（fn_milestone_delivery_risk）：连接 milestones、milestone_nodes、requirements、requirement_test_links、test_cases、requirement_links、defects、branches、change_sets，统计各里程碑的未完成需求、测试缺口、依赖阻塞、未关闭缺陷与待合入变更，用于发布前风险评估 |

### FR-15 性能分析实验室

| 编号    | 需求描述                                                                               |
| ------- | -------------------------------------------------------------------------------------- |
| FR-15.1 | 提供索引性能对比功能，可对比有索引与无索引场景下的查询耗时                             |
| FR-15.2 | 提供视图性能对比功能，可对比使用视图与直接 SQL 查询的执行效率                          |
| FR-15.3 | 支持生成基准测试数据（通过 generate_benchmark_data.sql），用于大数据量场景下的性能验证 |

## 2.3 非功能性需求

### NFR-1 数据完整性

| 编号    | 需求描述                                                                                                |
| ------- | ------------------------------------------------------------------------------------------------------- |
| NFR-1.1 | **实体完整性**：所有 15 张关系表均定义主键，确保每条记录可唯一标识                                      |
| NFR-1.2 | **参照完整性**：通过 22 条外键约束维护实体间的引用关系，采用 CASCADE / SET NULL / RESTRICT 混合删除策略 |
| NFR-1.3 | **域完整性**：通过 17 条 CHECK 约束限制状态、角色、优先级、类型等字段的取值范围                         |
| NFR-1.4 | **业务完整性**：通过 8 个触发器实现跨表范围校验（同项目约束）、层级规则校验和循环引用检测               |
| NFR-1.5 | **应用层校验**：后端接口对用户输入进行参数校验，违规操作时返回明确的中文错误提示信息                    |

### NFR-2 性能需求

| 编号    | 需求描述                                                                                                 |
| ------- | -------------------------------------------------------------------------------------------------------- |
| NFR-2.1 | 为高频查询场景设计了 35+ 个索引，覆盖所有表的外键列、状态列和常用筛选列                                  |
| NFR-2.2 | 对稀疏列（如 assignee、parent_id、product_id）采用条件索引（WHERE ... IS NOT NULL），减少索引存储开销    |
| NFR-2.3 | 提供 2 个数据库视图（v_requirement_details、v_project_statistics），预聚合常用统计数据，避免重复多表连接 |
| NFR-2.4 | 支持万级数据量下的基准性能测试，验证索引与视图对查询效率的提升效果                                       |

### NFR-3 可扩展性

| 编号    | 需求描述                                                                                                                          |
| ------- | --------------------------------------------------------------------------------------------------------------------------------- |
| NFR-3.1 | 需求、里程碑、分支、审计日志等实体使用 JSONB 字段存储标签、元数据、自定义字段和详情数据，可在不修改表结构的前提下扩展新的业务属性 |
| NFR-3.2 | 评论和审计日志采用多态关联模式（target_type + target_id），可在不新增外键的情况下扩展支持新的目标对象类型                         |

### NFR-4 可用性

| 编号    | 需求描述                                                                                             |
| ------- | ---------------------------------------------------------------------------------------------------- |
| NFR-4.1 | 前端采用响应式布局，提供侧边栏导航、模态框表单和数据表格等标准交互组件                               |
| NFR-4.2 | 数据库操作错误（如违反唯一约束、外键约束、非空约束等）在交互界面以中文提示展示，帮助用户理解问题原因 |
| NFR-4.3 | 前端使用 Claude 设计系统风格，具备现代化的扁平极简视觉风格                                           |

### NFR-5 架构与技术约束

| 编号    | 需求描述                                                                             |
| ------- | ------------------------------------------------------------------------------------ |
| NFR-5.1 | 采用前后端分离架构，后端提供 RESTful API，前端通过 HTTP 请求与后端通信               |
| NFR-5.2 | 后端基于 FastAPI 框架，支持 CORS 跨域配置                                            |
| NFR-5.3 | 前端基于 Vue.js 3 Composition API + Vite 构建工具链                                  |
| NFR-5.4 | 数据库使用 PostgreSQL，利用其 JSONB、条件索引、WITH RECURSIVE、FILTER 聚合等高级特性 |
