# 3.6 程序实现

## 3.6.1 系统功能架构

系统采用前后端分离架构，后端基于 Python FastAPI 框架提供 RESTful API，前端基于 Vue.js 3 构建单页应用，数据库使用 PostgreSQL。

系统核心模块划分如下：

| 模块 | 说明 |
|------|------|
| FR-1 数据库连接管理 | 自动连接检测、连接池管理 |
| FR-2~FR-4 产品/项目/需求管理 | CRUD 操作，含树形层级、状态流转 |
| FR-5~FR-6 测试用例与缺陷管理 | 多对多关联、复合外键约束 |
| FR-7~FR-8 里程碑与分支变更管理 | 快照、基线、变更集追踪 |
| FR-9~FR-10 评论与审计日志 | 多态关联、完整性校验 |
| FR-11~FR-13 SQL 编辑器与统计 | 任意 SQL 执行、统计视图封装 |
| FR-14~FR-15 复杂查询与性能分析 | 存储函数、索引对比验证 |

从程序实现的分层来看，系统按 **表示层 → 接口层 → 数据访问层 → 数据库对象层** 逐层拆分：

| 实现层次 | 主要文件/对象 | 作用 |
|----------|---------------|------|
| 前端表示层 | `frontend/src/views/*.vue`、`frontend/src/components/*.vue` | 提供表格、树形视图、表单弹窗、统计展示等交互界面 |
| 前端 API 封装层 | `frontend/src/api/index.js` | 统一封装 Axios 请求，屏蔽具体 URL 和 HTTP 方法细节 |
| 后端接口层 | `backend/routers/connection.py`、`query.py`、`crud.py`、`extended_crud.py`、`statistics.py` | 按业务域暴露 RESTful API，处理参数校验和错误翻译 |
| 数据访问层 | `backend/database.py` 中的 `DatabaseManager` | 负责连接池、SQL 执行、结果序列化、事务清理 |
| 启动与 schema 自修复层 | `backend/local_postgres.py`、`backend/schema_bootstrap.py` | 负责本地 PostgreSQL 自启动、核心 schema 检查与补丁修复 |
| 数据库对象层 | 15 张表、8 个触发器、2 个视图、3 个复杂查询函数 | 负责最终的数据持久化、完整性约束和复杂统计 |

## 3.6.3 关键数据结构和实体

数据库共包含 **15 张关系表**，按业务域分为七层：

### 一、产品层（Product Layer）

| 表名 | 主键 | 说明 |
|------|------|------|
| manage_products | product_id | 产品信息，含状态、版本、路线图、JSONB 标签 |
| manage_product_members | id (SERIAL) | 产品成员，UK(product_id, user_id) 保证同一用户不重复加入 |

### 二、项目层（Project Layer）

| 表名 | 主键 | 说明 |
|------|------|------|
| manage_projects | project_id | 项目，含 FK→products(ON DELETE SET NULL)，支持脱离产品独立存在 |
| manage_project_members | id (SERIAL) | 项目成员，UK(project_id, user_id) |

### 三、需求层（Requirements Layer）

| 表名 | 主键 | 关键设计 |
|------|------|----------|
| manage_requirements | req_id | 树形结构：parent_id 自引用 FK(ON DELETE SET NULL)，三层类型 top_level/low_level/task，deleted 软删除，JSONB 自定义字段 |
| manage_requirement_links | link_id (SERIAL) | M:N 自关联（blocks/depends_on/relates_to/duplicates），UK(source, target, type)，CHK 不自引用 |

### 四、测试与缺陷层

| 表名 | 主键 | 关键设计 |
|------|------|----------|
| manage_test_cases | test_case_id | UK(project_id, test_case_id)，归属项目 |
| manage_requirement_test_links | link_id (SERIAL) | M:N 关联表，UK(requirement_id, test_case_id) |
| manage_defects | defect_id | **复合外键** FK(project_id, requirement_id) → manage_requirements(project_id, req_id)，保证缺陷与需求同项目 |

### 五、里程碑层

| 表名 | 主键 | 关键设计 |
|------|------|----------|
| manage_milestones | milestone_id | UK(project_id, milestone_id)，里程碑类型 regular/baseline/branch/merge，is_baseline 标记基线 |
| manage_milestone_nodes | snapshot_id (TEXT) | 需求快照，FK→milestones(ON DELETE CASCADE)，FK→requirements(ON DELETE CASCADE)，JSONB 存储完整快照数据 |

### 六、分支与变更层

| 表名 | 主键 | 关键设计 |
|------|------|----------|
| manage_branches | branch_id | UK(project_id, name)，FK→milestones(ON DELETE **RESTRICT**)，基线被引用时禁止删除 |
| manage_change_sets | change_id | FK→branches(ON DELETE CASCADE)，FK→requirements(ON DELETE SET NULL)，JSONB 存储变更前后数据 |

### 七、协同与审计层

| 表名 | 主键 | 关键设计 |
|------|------|----------|
| manage_comments | comment_id | **多态关联**：target_type+target_id 可指向 requirement/defect/test_case/milestone；该关联不是数据库原生外键，而是通过 target_type + target_id 配合触发器完成完整性校验；reply_to_id 自引用构成评论树；同一项目约束 |
| manage_audit_logs | log_id | 多态关联 product_id+project_id，JSONB 存储 detail，actor 记录操作人 |

### 八、应用层核心数据结构

除了数据库表结构外，程序实现中还有若干直接支撑运行时行为的内存数据结构：

| 数据结构 | 所在位置 | 关键字段 | 作用 |
|----------|----------|----------|------|
| `DatabaseManager` | `backend/database.py` | `_pool`、`_lock`、`_idle`、`_active_ops`、`_info` | 维护线程安全连接池，控制连接借还和断开时机 |
| 查询返回结构 | `DatabaseManager.execute()` | `type`、`columns`、`rows`、`row_count` | 统一封装 SELECT 与非 SELECT 的返回格式，便于前端表格直接渲染 |
| Pydantic 请求模型 | `crud.py` / `extended_crud.py` / `connection.py` / `query.py` | 如 `ConnectRequest`、`RequirementCreate`、`CommentCreate` | 将 HTTP JSON 请求映射为强类型对象，承担接口层第一道校验 |
| 前端页面状态对象 | 各 `*.vue` 页面中的 `ref/reactive` | `items`、`loading`、`error`、`form`、`expanded` | 保存页面数据、表单数据、弹窗状态和树形展开状态 |
| 需求树节点结构 | `RequirementsView.vue` 中 `buildTree()` 结果 | `req_id`、`parent_id`、`depth`、`children` | 将数据库中的平面需求列表重建为树形结构以支持层级展示 |

其中，`DatabaseManager` 是后端最核心的运行时对象，其结构可以概括为：

```python
class DatabaseManager:
    _pool: ThreadedConnectionPool | None   # 数据库连接池
    _lock: threading.Lock                  # 保护共享状态
    _idle: threading.Condition             # disconnect 时等待活跃操作归还连接
    _active_ops: int                       # 当前借出的连接操作数
    _info: Dict[str, Any]                  # 当前连接信息
    _pool_config = {"minconn": 1, "maxconn": 8}
```

该设计避免了“每次请求新建连接”的高开销问题，也避免了关闭连接池时仍有线程在执行 SQL 的竞态风险。

## 3.6.4 关键函数与触发器实现

### 3.6.4.1 业务完整性触发器（8 个）

触发器统一采用 `BEFORE INSERT OR UPDATE OF 指定列 ON 表名 FOR EACH ROW` 模式，校验失败时通过 `RAISE EXCEPTION` 中止操作并返回中文错误信息：

| 函数名 | 触发定义 | 校验内容 |
|--------|----------|----------|
| fn_validate_requirement_hierarchy | `BEFORE INSERT OR UPDATE OF project_id, requirement_type, parent_id ON manage_requirements FOR EACH ROW` | 层级规则（top/low/task）、同项目约束、自引用检查；循环引用检测仅在 UPDATE 时通过递归 CTE 执行 |
| fn_validate_requirement_link_scope | `BEFORE INSERT OR UPDATE OF source_req_id, target_req_id ON manage_requirement_links FOR EACH ROW` | 源/目标需求存在且同项目、被删除需求禁止关联 |
| fn_validate_requirement_test_link_scope | `BEFORE INSERT OR UPDATE OF requirement_id, test_case_id ON manage_requirement_test_links FOR EACH ROW` | 需求与测试用例必须同项目 |
| fn_validate_branch_scope | `BEFORE INSERT OR UPDATE OF project_id, base_milestone_id ON manage_branches FOR EACH ROW` | 分支与基线里程碑必须同项目 |
| fn_validate_milestone_node_scope | `BEFORE INSERT OR UPDATE OF milestone_id, requirement_id ON manage_milestone_nodes FOR EACH ROW` | 里程碑节点与需求必须同项目 |
| fn_validate_change_set_scope | `BEFORE INSERT OR UPDATE OF branch_id, requirement_id ON manage_change_sets FOR EACH ROW` | 变更集与需求必须同项目（requirement_id 为空时跳过） |
| fn_validate_comment_scope | `BEFORE INSERT OR UPDATE OF project_id, target_type, target_id, reply_to_id ON manage_comments FOR EACH ROW` | 评论目标存在、目标与评论同项目、回复同项目约束 |
| fn_validate_audit_log_scope | `BEFORE INSERT OR UPDATE OF project_id, product_id ON manage_audit_logs FOR EACH ROW` | project 与 product 归属一致性校验 |

**循环父子关系检测核心逻辑**（fn_validate_requirement_hierarchy）：

```sql
-- UPDATE 时检测：若在祖先链表中发现自己的 req_id，说明形成循环
WITH RECURSIVE ancestors AS (
    SELECT req_id, parent_id
      FROM manage_requirements
     WHERE req_id = NEW.parent_id          -- 从父节点开始
    UNION ALL
    SELECT r.req_id, r.parent_id
      FROM manage_requirements r
      JOIN ancestors a ON r.req_id = a.parent_id
     WHERE a.parent_id IS NOT NULL
)
SELECT 1 FROM ancestors WHERE req_id = NEW.req_id LIMIT 1
-- 若存在结果 → RAISE EXCEPTION '检测到循环父子关系'
```

### 3.6.4.2 视图（2 个）

**视图 1：v_requirement_details（需求完整视图）**

该视图为普通 `VIEW`，通过两个 CTE 封装测试用例数量和缺陷数量的聚合逻辑，减少应用层重复编写复杂 SQL：

```sql
WITH test_stats AS (
    SELECT requirement_id, COUNT(DISTINCT test_case_id) AS test_case_count
    FROM manage_requirement_test_links GROUP BY requirement_id
),
defect_stats AS (
    SELECT requirement_id,
           COUNT(DISTINCT defect_id) AS defect_count,
           COUNT(DISTINCT CASE WHEN status IN ('open','in_progress') THEN defect_id END) AS open_defect_count
    FROM manage_defects GROUP BY requirement_id
)
SELECT r.req_id, r.title, r.requirement_type, r.status, r.priority, r.assignee,
       p.name AS project_name,
       COALESCE(ts.test_case_count,0) AS test_case_count,
       COALESCE(ds.defect_count,0) AS defect_count,
       COALESCE(ds.open_defect_count,0) AS open_defect_count,
       CASE WHEN COALESCE(ts.test_case_count,0) > 0 THEN TRUE ELSE FALSE END AS has_test_coverage
FROM manage_requirements r
JOIN manage_projects p ON p.project_id = r.project_id
LEFT JOIN test_stats ts ON ts.requirement_id = r.req_id
LEFT JOIN defect_stats ds ON ds.requirement_id = r.req_id
WHERE r.deleted = FALSE;
```

**视图 2：v_project_statistics（项目统计视图）**

该视图为普通 `VIEW`，通过五个 CTE 分别统计需求、缺陷、测试用例、里程碑、分支等指标，将项目级统计逻辑统一封装，减少应用层重复拼接复杂 SQL：

```sql
WITH req_stats AS (
    SELECT project_id,

),
def_stats AS (
    SELECT project_id, COUNT(*) AS total_defects,
           COUNT(*) FILTER (WHERE severity='critical') AS critical_defects,
           COUNT(*) FILTER (WHERE status IN ('open','in_progress')) AS open_defects
    FROM manage_defects GROUP BY project_id
),
tc_stats AS (...),
ms_stats AS (...),
br_stats AS (...)
SELECT p.project_id, p.name, r.total_requirements, r.completed_count,
       ROUND((r.completed_count::NUMERIC / r.total_requirements)*100, 2) AS completion_rate_percent,
       d.total_defects, tc.total_test_cases, m.total_milestones, b.total_branches, ...
FROM manage_projects p
LEFT JOIN req_stats r ON r.project_id = p.project_id
LEFT JOIN def_stats d ON d.project_id = p.project_id
...;
```

### 3.6.4.3 复杂查询函数（3 个）

**函数 1：fn_requirement_trace（需求追溯查询）**

连接 5 张表（requirements、projects、requirement_test_links、test_cases、defects），使用 2 个并列 CTE 分别聚合测试用例和缺陷的 JSONB 数组，返回需求的完整追溯信息。其关键技术点包括：

- `JSONB_AGG(DISTINCT JSONB_BUILD_OBJECT(...)) FILTER (...)`：将测试用例和缺陷聚合为 JSONB 数组，并去除重复项、过滤空值
- `COALESCE(..., '[]'::JSONB)`：在无关联数据时返回空 JSON 数组，而不是 `NULL`
- 返回列名采用函数定义中的精确名称：`requirement_title`、`requirement_type`

```sql
RETURN QUERY
WITH test_case_agg AS (
    SELECT rtl.requirement_id,
           JSONB_AGG(DISTINCT JSONB_BUILD_OBJECT(
               'test_case_id', tc.test_case_id,
               'title', tc.title, 'status', tc.status, 'link_type', rtl.link_type
           )) FILTER (WHERE tc.test_case_id IS NOT NULL) AS test_cases,
           COUNT(DISTINCT tc.test_case_id) AS test_case_count
    FROM manage_requirement_test_links rtl
    LEFT JOIN manage_test_cases tc ON tc.test_case_id = rtl.test_case_id
    GROUP BY rtl.requirement_id
),
defect_agg AS (
    SELECT df.requirement_id,
           JSONB_AGG(DISTINCT JSONB_BUILD_OBJECT(
               'defect_id', df.defect_id, 'title', df.title,
               'severity', df.severity, 'priority', df.priority,
               'status', df.status, 'current_assignee', df.current_assignee
           )) FILTER (WHERE df.defect_id IS NOT NULL) AS defects,
           COUNT(DISTINCT df.defect_id) AS total_defect_count,
           COUNT(DISTINCT CASE WHEN df.status IN ('open','in_progress') THEN df.defect_id END) AS open_defect_count
    FROM manage_defects df GROUP BY df.requirement_id
)
SELECT req.req_id, req.title AS requirement_title, req.requirement_type, req.status, req.priority, req.assignee,
       proj.name AS project_name, COALESCE(tca.test_cases,'[]'::JSONB), COALESCE(da.defects,'[]'::JSONB),
       COALESCE(tca.test_case_count,0), COALESCE(da.total_defect_count,0), COALESCE(da.open_defect_count,0)
FROM manage_requirements req
JOIN manage_projects proj ON proj.project_id = req.project_id
LEFT JOIN test_case_agg tca ON tca.requirement_id = req.req_id
LEFT JOIN defect_agg da ON da.requirement_id = req.req_id
WHERE req.deleted = FALSE AND req.project_id = p_project_id
ORDER BY req.created_at DESC;
```

**函数 2：fn_project_progress（项目进度统计）**

使用 **4 个并列 CTE** 分别统计需求进度、缺陷、测试覆盖率、里程碑，最终通过 `LEFT JOIN` 汇总到项目维度，返回 **16 个字段**：

- `requirement_stats` — 需求总数、各状态数量
- `defect_stats` — 缺陷总数、critical 数量、open 数量
- `test_coverage_stats` — 有测试用例的需求数量（EXISTS 子查询）
- `milestone_stats` — 里程碑总数、基线数量

最终 SELECT 将四个 CTE 的结果通过 `LEFT JOIN` 汇总到项目维度，并计算完成率和测试覆盖率。

**函数 3：fn_milestone_delivery_risk（里程碑交付风险查询）**

关联里程碑、快照节点、需求、需求关联、测试、缺陷、分支、变更和项目等多张业务表，使用 **5 个 CTE** 分层聚合里程碑粒度的风险指标：

1. `milestone_requirement_stats` — 范围内需求数、未完成数、无测试覆盖数、被阻塞数；其中“被阻塞”只检查当前需求是否**直接依赖**一个未完成需求，“无测试覆盖”只统计是否存在 `active` 状态测试用例
2. `milestone_defect_stats` — 未关闭缺陷数、critical 缺陷数
3. `milestone_branch_stats` — 活跃分支数、待合入变更数、最新活动时间
4. `risk_metrics` — 基于加权公式计算 risk_score：
   ```
   risk_score = incomplete * 2.0 + uncovered * 1.5 + blocked * 2.5
              + unresolved_defects * 2.0 + critical_defects * 3.0
              + pending_changes * 0.5
   ```
5. 最终 SELECT — 关联里程碑元数据，risk_level 根据 score 分为 high(≥12) / medium(≥6) / low

### 3.6.4.4 应用层关键函数（补充）

除数据库函数和触发器外，应用层也实现了若干直接决定系统行为的关键函数，它们构成了“接口层校验 + 数据访问层封装 + 前端状态转换”的主干逻辑。

**函数 1：`DatabaseManager.connect()`（连接建立与 schema 引导）**

该函数串联了本地服务拉起、连接池建立、版本探测和 schema 修复四个步骤：

1. `ensure_local_postgres_ready(host, port)`：若目标是本机 PostgreSQL，则尝试自动启动数据库服务
2. `ThreadedConnectionPool(...)`：建立最小 1、最大 8 的连接池
3. `SELECT version()`：验证连接真实可用
4. `ensure_requirements_schema(test_conn)`：检查 15 张核心表和运行时函数是否完整，必要时执行 `requirements_db.sql` 与约束补丁

这使得系统具备“第一次运行即可自动补齐环境”的能力，降低了手工初始化成本。

**函数 2：`DatabaseManager.execute()`（统一 SQL 执行入口）**

该函数承担后端几乎全部 SQL 的实际执行工作，核心实现特征包括：

- 使用 `RealDictCursor` 返回字典行，减少字段名到 JSON 的手动映射
- 对 `datetime`、`Decimal`、`UUID`、`bytes` 等 PostgreSQL 类型进行可序列化转换
- 对查询语句返回 `columns + rows`
- 对写入语句返回 `OK — n row(s) affected`
- 对 `IntegrityError` 做统一中文翻译

其返回结构如下：

```python
{
    "type": "result" | "command",
    "columns": [...],
    "rows": [...],
    "row_count": n
}
```

**函数 3：`validate_requirement_parent()`（需求层级预校验）**

该函数作为数据库触发器之前的应用层前置守卫，其逻辑与数据库层规则保持一致，能在请求到达时提前给出更友好的错误信息：

```python
def validate_requirement_parent(project_id, requirement_type, parent_id, current_req_id=None):
    if not parent_id and requirement_type != "top_level":
        raise HTTPException(...)
    if requirement_type == "top_level" and parent_id:
        raise HTTPException(...)
    if current_req_id is not None and parent_id == current_req_id:
        raise HTTPException(...)
    parent = get_requirement_record(parent_id)
    # 校验父需求存在、同项目、父类型合法
    # UPDATE 时再用递归 CTE 检测环
```

这种实现使前端在“保存前”即可拿到明确错误，而不是只看到数据库报错原文。

**函数 4：`handle_db_error()`（数据库异常翻译）**

该函数把底层异常统一转换为适合界面展示的中文信息，例如唯一约束、外键约束、非空约束和 schema 未初始化等问题。这样前端只需要读取 `detail` 字段，不必理解 PostgreSQL 原始错误文本。

**函数 5：`buildTree()`（前端需求树重建）**

数据库中的需求记录是扁平表结构，而树形视图依赖前端进行二次组装：

```javascript
function buildTree(flat, parentId = null, depth = 0) {
  return flat
    .filter(r => r.parent_id === parentId)
    .map(r => ({
      ...r,
      depth,
      children: buildTree(flat, r.req_id, depth + 1),
    }))
}
```

该函数把关系表中的 `parent_id` 递归转换为页面可直接渲染的 `children` 结构，使系统同时支持“表格视图”和“树形视图”两种展示模式。

**函数 6：`create_milestone_node()`（快照写入）**

该函数体现里程碑模块的核心实现思路：将需求状态复制到 `manage_milestone_nodes`，其中 `snapshot_data` 采用 JSONB 保存完整快照。这为后续差异对比、分支变更和交付风险分析提供了静态历史基线。

## 3.6.5 索引设计

系统中显式 `CREATE INDEX` 创建的索引共有 **47 个**，此外若干 `UNIQUE` 约束还会隐式创建唯一索引。索引覆盖所有高频查询场景，稀疏列采用条件索引减少存储开销：

| 索引类型 | 示例 | 说明 |
|----------|------|------|
| 外键列索引 | idx_requirements_project, idx_defects_requirement | 加速 JOIN |
| 状态列索引 | idx_requirements_status, idx_defects_status | 加速条件筛选 |
| 条件索引（稀疏列） | `WHERE assignee IS NOT NULL` | 只索引有值的行，减少索引体积 |
| 唯一索引 | UK(product_id, user_id) | 同时保证唯一性和查询性能 |
| 复合索引 | (target_type, target_id) | 多态查询覆盖 |

## 3.6.6 应用层错误处理

系统采用“双层完整性保障”机制：应用层通过 Pydantic 模型和显式参数校验提前拦截非法输入，数据库层通过主键、外键、唯一约束、非空约束、检查约束和触发器保证最终一致性。对于仍然由数据库层捕获的完整性错误，`DatabaseManager.execute()` 会统一翻译为中文提示返回给前端：

```python
except pg_errors.IntegrityError as exc:
    msg = str(exc)
    if "unique" in msg.lower() or "duplicate" in msg.lower():
        raise RuntimeError("违反唯一约束：" + msg)
    if "foreign key" in msg.lower():
        raise RuntimeError("外键引用无效：" + msg)
    if "not-null" in msg.lower():
        raise RuntimeError("字段不能为空：" + msg)
    if "check" in msg.lower():
        raise RuntimeError("数据校验失败：" + msg)
    raise RuntimeError("数据完整性错误：" + msg)
```
